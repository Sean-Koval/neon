================================================================================
SUITES LIST — app/suites/page.tsx
================================================================================
Evaluation suite definitions. Promoted from hidden to visible nav under EVALUATE.
A suite is a reusable collection of test cases + scorers targeting a specific agent.
Route: /suites

DESIGN NOTES:
  1. Suites are the entry point to the evaluation lifecycle. The list view must
     surface enough context to decide which suite to run next — last run result,
     case count, scorer coverage, and pass-rate trend.
  2. "Create Suite" links to /suites/new (existing 4-step wizard), not a dialog.
     The wizard handles the complexity of multi-case creation with JSON input
     editors, scorer configuration, and execution settings.
  3. Cards are links to /suites/[id]. "Run This Suite" is a secondary action
     that opens the existing StartEvalRunDialog pre-filled with the suite.
  4. Summary stats strip aggregates across all suites for a project-level view.
  5. Filters are URL-synced via shallow routing (searchParams).
  6. ScorerType values: tool_selection, reasoning, grounding, efficiency, custom.
  7. Suite data is cached for 5 minutes (staleTime). No real-time polling needed.

DATA: trpc.suites.list, trpc.evals.listRuns (for last-run stats per suite)

DATA SOURCES:
  - trpc.suites.list → EvalSuite[] (suite metadata + cases array)
  - trpc.evals.listRuns({ suiteId }) → EvalRun[] (for last-run stats per suite)
  - Last-run stats are derived client-side: take most recent run per suite,
    extract summary.passed / summary.total_cases and summary.avg_score
  - Summary stats strip is computed by aggregating across all suites
  - Agent list for filter dropdown is derived from unique agent_id values
  - StartEvalRunDialog is the existing component from eval-runs page
  - Suite creation wizard is at /suites/new (existing 4-step flow)

BEHAVIORAL NOTES:
  1. Cards are clickable links to /suites/[id], but the "Run Suite" button
     is a separate click target that opens StartEvalRunDialog (stopPropagation).
  2. Search is client-side (debounced 300ms) since suite count is typically
     low (< 50). No server-side pagination needed.
  3. Sort persists in URL searchParams. Default: most recently run first.
  4. Suites without any runs sort to the bottom in the default sort order.
  5. The "Run Suite" button pre-fills StartEvalRunDialog with:
     { suiteId, agentId (from suite), defaultScorers (from suite) }
  6. Scorer badges use abbreviated labels: tool_selection -> "Tool",
     reasoning -> "Reason", grounding -> "Ground", efficiency -> "Effic",
     custom -> "Custom".
  7. Cache: staleTime 5 minutes. No refetchInterval (suites change rarely).
  8. Grid is responsive: 1 col on mobile, 2 on md, 3 on xl.
