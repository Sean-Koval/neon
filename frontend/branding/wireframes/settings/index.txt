SETTINGS â€” app/settings/page.tsx
================================================================================
Central configuration for project, authentication, LLM providers, and
infrastructure health. The only page in the sidebar not grouped under
Monitor/Evaluate/Improve â€” sits below the divider as a utility page.
Route: /settings

PARENT: Root layout with sidebar
DATA: useProjectSettings() â†’ GET /api/settings
      useAuth() â†’ sessionStorage + NEXT_PUBLIC_API_KEY
      useLlmProviders() â†’ GET /api/settings/llm-providers
      useInfrastructureHealth() â†’ GET /api/settings/health (30s auto-refresh)
LAYOUT: p-6, max-w-4xl, space-y-6

DESIGN NOTES:

1. Settings is a low-traffic, high-trust page. Users visit to configure
   once and check infrastructure health occasionally. The design
   prioritizes clarity and confidence â€” every action is explicit,
   every state is visible, and destructive actions have confirmation.

2. max-w-4xl constraint is intentional. Forms and configuration
   panels don't benefit from full-width layouts â€” they become
   harder to scan. The constrained width keeps label-value pairs
   readable and gives the page a focused, professional feel.

3. Four tabs map to four concerns:
   - Project: identity + defaults (who am I, what are my baselines)
   - API Keys: authentication (how do SDKs connect to me)
   - LLM Providers: model access (which models can I use)
   - Infrastructure: system health (are my services running)

4. Project tab splits into two cards: read-only System Info (env-
   driven, can't change in UI) and editable Project Defaults
   (score thresholds, SLA targets, default model). This separation
   makes it clear which values are configurable vs fixed.

5. API Keys uses a client-side sessionStorage model (not server-
   managed key lifecycle) because Neon's auth is workspace-scoped
   via the withAuth middleware. The key format ae_<env>_<key>
   encodes environment in the prefix. The wireframe documents
   the current model faithfully â€” server-managed CRUD would
   require backend changes not in scope.

6. LLM Providers are read-only status checks (env var existence).
   This is intentional for security â€” API keys for LLM providers
   are server-side secrets, not client-configurable. The UI shows
   what's available, not how to configure it.

7. Infrastructure auto-refreshes every 30 seconds. The Refresh
   button provides manual override. The overall status badge
   (healthy/degraded/unhealthy) answers "is my system OK?" in
   one glance. Service cards drill into specifics.

8. No Danger Zone in this version. Project deletion and data
   export are admin operations that belong in a future admin
   panel, not in per-project settings. Avoids scope creep.

9. Tab icons provide quick visual scanning without reading labels.
   Building2 (project), Key (auth), Server (providers), Database
   (infrastructure) â€” all from lucide-react, consistent with
   the rest of the application.

10. The page uses standard form patterns: labels above inputs,
    helper text below, inline validation errors, success feedback
    that auto-clears. Nothing novel â€” Settings should feel
    familiar and trustworthy.

===============================================================================
BEHAVIORAL NOTES
===============================================================================

1. Tab state stored in URL: &tab=project (default) / api-keys / llm /
   infrastructure. Shallow routing via router.replace so the page
   doesn't fully re-render on tab switch.

2. Project tab System Information is fully read-only. Values come
   from environment variables at server startup. The "Contact your
   administrator" note sets correct expectations about mutability.

3. Evaluation Defaults card tracks dirty state. Save button is
   disabled until at least one value differs from the server state.
   On save, button shows spinner. On success, toast notification.
   On error, inline error message below button.

4. API key is stored in sessionStorage (not localStorage) for
   security. It can also be set via NEXT_PUBLIC_API_KEY env var.
   Environment variable takes priority on initial load.

5. API key validation is functional, not just format-based. After
   format check (ae_<env>_<key>), the key is tested by calling
   api.getSuites(). If the call fails, the key is cleared and an
   error is shown. This catches expired or revoked keys.

6. "Clear API key" is not a button â€” it's a text link styled in
   red. This de-emphasizes the destructive action while keeping
   it accessible. No confirmation dialog needed â€” re-entering
   the key is trivial.

7. LLM Providers tab is informational only. The server checks
   env var existence (!!process.env.ANTHROPIC_API_KEY) and
   returns boolean status. No secrets are transmitted to the
   client. The cards show whether providers are available, not
   their configuration.

8. Infrastructure health auto-refreshes every 30 seconds via
   React Query's refetchInterval. The Refresh button provides
   manual override. The RefreshCw icon spins (animate-spin)
   while isFetching is true.

9. Infrastructure Overall Status uses a 3-tier model:
   healthy (all services connected), degraded (partial),
   unhealthy (none). Badge color matches the tier. The
   "Last checked" timestamp uses health.timestamp formatted
   via toLocaleString().

10. Connection URLs (ClickHouse URL, Temporal address) are shown
    as font-mono text below the service name. They're only
    rendered if the health response includes them (conditional).
    These help administrators verify they're connecting to the
    right endpoints.

11. The max-w-4xl constraint on the page means Settings doesn't
    stretch to full width on large screens. This is intentional
    for form readability. All other pages (Command Center, Agents,
    Traces, etc.) use full width.

12. Success feedback auto-clears: API key save confirmation
    disappears after 3 seconds (CONFIG.SUCCESS_FEEDBACK_MS).
    Project defaults save uses a toast that auto-dismisses.
    No persistent success states.
